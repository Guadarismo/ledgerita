## 3. Cambios implementados hoy

### 3.1 Wallet persistente (`src/crypto.rs`)
- **Archivo**: `~/.ledgerita/wallet.key` (formato hex, 64 caracteres)
- **Semilla**: 32 bytes generados con `getrandom` en primera ejecución
- **Derivación**: La misma semilla crea:
  - `ed25519_dalek::SigningKey` → para firmar transacciones (wallet)
  - `libp2p::identity::Keypair` → para identidad de red P2P
- **Funciones nuevas**:
  - `load_or_create_wallet(data_dir)` → carga o crea wallet, devuelve `(SigningKey, Keypair)`
  - `show_address(data_dir)` → solo lee la dirección pública sin exponer la llave privada
  - `Address::from_signing_key()` → constructor desde SigningKey
  - `Address::to_hex()` → representación hex de la dirección
  - Helper `hex_encode()` / `hex_decode()` (sin dependencias externas)

### 3.2 Conexión activa entre pares (`src/network.rs`)
- `run_p2p_node(local_key, connect_peers)` ahora recibe:
  - `identity::Keypair` → identidad persistente (no se genera aleatoria)
  - `Vec<Multiaddr>` → lista de pares a conectar activamente
- Después de `listen_on()`, itera `connect_peers` y llama `swarm.dial(addr)`
- Nuevos eventos en el loop principal:
  - `ConnectionEstablished { peer_id }` → ✅
  - `ConnectionClosed { peer_id, cause }` → 🔴
  - `OutgoingConnectionError { peer_id, error }` → ⚠️

### 3.3 CLI y orquestación (`src/main.rs`)

| Flag | Alias | Descripción |
|------|-------|-------------|
| `--connect` | `-c` | Dirección multiaddr de pares a conectar (repetible) |
| `--data-dir` | `-d` | Directorio de datos (default: `~/.ledgerita/`) |
| `--show-address` | `-s` | Solo imprime dirección pública sin iniciar nodo |

Default `data_dir`:
```rust
fn default_data_dir() -> PathBuf {
    let home = std::env::var("HOME")
        .or_else(|_| std::env::var("USERPROFILE"))
        .unwrap_or_else(|_| ".".to_string());
    PathBuf::from(home).join(".ledgerita")
}
```

### 3.4 Dependencia agregada
- `getrandom = "0.2"` — generación segura de semilla de 32 bytes (presente en `local_deps/`)

## 5. Pendientes para futuras sesiones

### 5.1 Seguridad — Crítico
- [ ] **Replay attacks**: el `nonce` existe en `Transaction` pero **nunca se valida** en `state.rs:apply_transaction()`. Hay que trackear el último nonce usado por cada cuenta y rechazar transacciones con nonce ≤ al actual.
- [ ] **Protección de wallet.key**: el archivo se guarda en texto plano. Ideal sería encriptarlo con una contraseña (por ahora es aceptable para desarrollo).

### 5.2 Interfaz de usuario — Wallet gráfica
- [ ] Construir HidroLedger: interfaz de usuario para:
  - Ver saldo de la wallet
  - Enviar LIT a otra dirección
  - Recibir LIT (generar dirección QR)
  - Ver historial de transacciones
- [ ] Opción: CLI interactiva con `rustyline` o UI web simple con `axum` + HTML/JS

### 5.3 Funcionalidad de red
- [ ] **Propagación de transacciones**: cuando un nodo recibe una transacción firmada válida, debe reenviarla a sus pares
- [ ] **Sincronización de estado**: al conectarse un nuevo nodo, debe descargar el state root y los balances actuales
- [ ] **Descubrimiento de pares**: mdns o bootstrap list para encontrar pares automáticamente
- [ ] **Persistencia de estado**: integrar Sled/RocksDB en lugar de mantener todo en memoria

### 5.4 Consenso y economía
- [ ] **Distribución de fees**: los fees se descuentan pero no se asignan a ningún validador (quemados). Implementar recompensa al creador del bloque.
- [ ] **Hidro-Checkpointing**: implementar el cierre de libro cada 2000 bloques con merkle root
- [ ] **Epoch rewards**: liberación del tesoro y distribución proporcional a nodos estables

### 5.5 Pruebas y validación

- [ ] Pruebas de estrés: muchas transacciones, múltiples pares conectándose
- [ ] Benchmark de rendimiento (tx/s, latencia de ping)

### 5.6 CI/Calidad

## 7. Arquitectura del protocolo

```
                    ┌──────────────────────┐
                    │      main.rs         │
                    │  CLI + Wallet load   │
                    └────────┬─────────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
       ┌──────────┐  ┌────────────┐  ┌──────────────┐
       │ crypto   │  │ transaction│  │   network    │
       │ .rs      │  │ .rs        │  │   .rs        │
       │ Address  │  │ Tx struct  │  │ Swarm P2P    │
       │ Wallet   │  │ sign/verify│  │ dial + ping  │
       │ key file │  │ serializ.  │  │ listeners    │
       └──────────┘  └─────┬──────┘  └──────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │   state.rs   │
                    │  BTreeMap    │
                    │  balances    │
                    │  state root  │
                    │  apply_tx()  │
                    └──────────────┘
```

### Flujo de una transacción
1. Usuario firma `Transaction` con su `SigningKey` (wallet)
2. Nodo valida firma → `is_signature_valid()` (Prueba de Fórmula)
3. Nodo aplica al `State` → `apply_transaction()` (descuenta saldo, acredita receptor)
4. State root se recalcula → `generate_state_root()` (SHA-256 del BTreeMap ordenado)
5. (Futuro) La transacción se propaga a pares vía P2P

### Identidad unificada
```
semilla 32 bytes → SHA-512 → clave ed25519
                     ├── SigningKey    → Wallet (transacciones)
                     └── Keypair P2P   → Nodo (PeerId)
```

## 8. Notas técnicas importantes

### Windows toolchain
- El proyecto usa dependencias vendorizadas en `local_deps/`
- `.cargo/config.toml` redirige crates.io → local_deps
- **Compila con MSVC** en Windows (requiere Visual Studio Build Tools)
- **NO compila con GNU** en Windows porque `ring` necesita `gcc.exe`
- En Linux (kettle02/03) se usó crates.io directamente (sin vendor) y compiló sin problemas

### Logs de errores históricos (resueltos)
Los archivos `.log` en el directorio raíz documentan la evolución:
- `error.log` — falta de `link.exe` (MSVC no instalado)
- `error_gnu.log` — `PublicKey` no existe en ed25519-dalek 2.x → cambiado a `VerifyingKey`
- `test_results.log` — `rand_core::OsRng` no disponible → eliminado, claves deterministas
- `compile_network.log` — `futures::StreamExt` no encontrado → cambiado a `libp2p::futures::StreamExt`
- `compile_network.log` — `ring` necesita gcc en GNU → usar MSVC en Windows
